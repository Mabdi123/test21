DDQ Picker Best Practices

  1. Optional Parameters - used by adding comments before and after each parameter you want to use as optional (variable not predefined). You must use each parameter name in the comments so that the parameters are cut once activating the query.

    If you want to have an optional alert_id you should write - /*alert_id_start*/ alert_id = alert_id /*alert_id_end*/.

  1. If you want a parameter that is located next to the 'where' keyword you have to include the 'where' inside the comment clause.

    select alert_id from alerts /*score_start*/ where score > score /*score_end*/

  1. If you want to have all of the parameters to be optional, there has to be a meaningless expression after the 'where' and only then you can insert the parameters:

    select alert_id alerts where 1=1 /*owner_id_start*/or owner_identifier = 'owner_id ' /*owner_id_end*/ /*date_start*/and alert_date > 'date' /*date_end*/ /*alert_type_start*/ or alert_type_id > alert_type /*alert_type_end*/ /*deleted_start*/ or deleted = deleted /*deleted_end*/

  1. When defining the UDDT in the DB, you have to define JavaScript (JS) parameters that are sent when activating the query in RCM. Each parameter has to be included in the JS string, even when using optional parameters. For example, if you composed a query that contains 10 different parameters where 3 of these parameters are optional then you should have the following string:

    function(){return '@@201003@@@@23.105@@@@@@@@%on%@@@@aaa@@@@@@@@12@@@@@@@@gcf@@@@dfgfdg@@'}

  1. Be aware that there are three different date formats used - one for the query saved in the DB, another for the picker window and a third format sent from the picker window to the Add/Edit window of each record.

Database Views

The RCM database Views expose internal RCM data for management and trade blotter reports. These views enable implementers to access RCM data, without the concern for possible changes within the data base structure.

These views will be materialized in your RCM database as a result of running the dbupgrade script during installation of RCM.

RCM contains two sets of views:

  • Identifier based views – Low performance views that exist due to backward compatibility
  • Internal join id based views – provide improved performance

The internal join id based views were introduced to make it easier for users to change names of objects and allow for duplicate object names.  Whenever a view of type (a) has a related, improved, type (b) view then they share the same name with the addition of the numeral 2.   When both types of views exist for an entity, such as v_acm_alert_audit and the improved v_acm_alert_audit2, .the internal join based view should be used. 

NOTE:

Join Ids are internal to a specific RCM repository instance. If you use the value of a specific Join Id in an SQL query and then move the query to a different RCM instance, you will get different results. Therefore, join Id fields should not be used in WHERE clauses to select rows from the repository. In other words:

Join ids are to be used ONLY for join, and MUST NOT be used against external databases or saved in any way.

They are not the same for two different RCM installations even when referring to the same logical object.

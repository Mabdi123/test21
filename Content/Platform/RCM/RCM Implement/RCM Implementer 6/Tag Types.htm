Tag Types

Tag Types provide the ability to connect notes via the RCM Alert types and Blotter Transaction Types by tagging them with the same value. Validation on the values of the tags is also defined by the implementer.

For example an account tag can include validation for a minimum length of 9 characters and a maximum length of 14 characters and it must begin with the letters abc (e.g., aac12345678). When the user adds the note for the relevant Alert Type or Transaction type an account tag appears. If the user enters a value for this tag then it must be entered according the defined validation settings. When the account number is entered correctly a tool tip indicates that the account number is valid.

If the account number is not valid then the input is underlined in red and a tool tip provides the reason for the invalid input.

To create a Tag type:

  1. Click the Create new Tag icon:
  1. Enter a display name and an identifier for this tag. The identifier is used as a reference ID for this Tag. Optionally, enter a description for this tag.

Tag Validations:

Enter the minimum and maximum characters for this tag. For example, if the tag is for an account number that is between 9 and 11 numbers then enter 9 for min length and 11 for max length.

  • You have the option to add a JavaScript function that verifies the tag’s value.

The JavaScript validation function accepts a single string argument (the tag value) and returns a string that stores the validation result. The validation result string should be an empty string when the validation is successful and an error message when it fails. The function definition should not include a function name. Click the more info link of the RCM Designer in order to view examples for JavaScript validation functions. If the user attempts to confirm an input that is not valid then a warning message appears and the note cannot be saved.

  • You have the option to add a drill down query (DDQ) for validating the data. The DDQ should receive one parameter and return two values: The parameter and the additional information can be a db column value or a concatenation of different columns. The possible scenarios for DDQ validation are the following:
  • Drill down query did not return any results. The value is underlined in yellow and a tooltip provides the reason for the invalid value.

The user has the option to confirm this value even when it is not validated by the DDQ.

  • Drill down query returned more than 1 result for the value. The value is underlined in yellow and a tooltip provides the reason for the invalid value.

The user has the option to confirm this value even when it is not validated by the DDQ.

  • DDQ result is valid: value is underlined in black and a tooltip indicates that the value is valid.

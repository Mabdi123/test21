v_acm_alerts2

Description: Returns the following fields for each Alert:

Field Name

Description

alert_join_id

The internal Join ID to the Alert

alert_id

The Identifier of the Alert.

Entity_id

The ID of the Entity

Entity_type_id

The type of the Entity

description

A description of the Item, if available. The description includes the first 4000 characters of the description.

description_full_text

A description of the Item, if available. The description includes the full text description (even if longer than 4000 characters).

alert_date

The Alert refers to this business date

create_date

The date that the Alert was created in the RCM database

alert_type_join_id

The Join ID to the Alert’s Alert Type (v_acm_alert_type2)

alert_status_join_id

The Join ID to the current step of the Alert (v_acm_alert_ statuses2)

score

The score assigned to the Alert

owner_join_id

The Join ID to the user who is responsible for this Alert (v_acm_users2)

bu_join_id

The Join ID to the current business unit with which this Alert is associated (v_acm_business_units2)

original_bu_join_id

The Join ID for the first business unit associated with this Alert (v_acm_business_units2)

fl_deleted

An indication whether the Alert was deleted

fl_archive

An indication whether the Alert was archived

fl_generated_by_ACM

An indication of whether the Alert was generated using the RCM application. E.g., manually generated Alerts or Alerts distributed by Web Services

fl_read

An indication whether the Alert was read

fl_read_by_owner

An indication whether the Alert was read by Alert owner

last_read_date

The last read date for the Alert

last_read_user_join_id

The Join ID to the user who last read this Alert (v_acm_users2)

last_update_date

The date that the Alert was last updated

last_update_user_join_id

The Join ID to the user who last updated this Alert (v_acm_users2)

closed_date

The date that the Alert was closed

alert_type_version

The Alert type version used by the system. Determines an alternative path for the required XSL file

fl_manual

An indication of whether this is a manually generated Alert

fl_incase

An indication of whether the alert been added to a case.

custom_attr_join_id

The Join ID to additional custom attributes for this Alert (v_acm_alert_custom_fields)

deadline_date

This field stores the date at which a work item will reach its deadline for processing. On that date or later the work item will perform escalation based on the workflow definition.

Both deadline_date and email_date actions are triggered by running a workflow utility that should run daily.

highlight_date

This field stores the work item highlight date. On that date or later the work item will show a UI indication on the work item grid.

email_date

This fields stores the work item email date. On that date or later an e-mail will be sent to the owner on that work item.

Both deadline_date and email_date actions are triggered by running a workflow utility that should run daily.

p11,p12…p50

Any custom fields defined for the associated Alert type

rfi_state

Indicates the status of RFIs within the Alert. It has one of three possible values:

0 – No RFIs

1 – Has RFIs but all messages have been read

2 – Has RFIs but there are unread messages

auto_esc_status_join_id

The status to be assigned to the work item if it auto escalates

global_auto_esc_status_join_id

The status to the assigned to the work item if it auto escalates based on a global action.

global_deadline_date

This field stores the global date at which a work item will reach its deadline for processing. On that date or later the work item will perform escalation based on the workflow definition no matter what the current status.

Both global_deadline_date and global_email_date actions are triggered by running a workflow utility that should run daily.

global_highlight_date

This field stores the global work item highlight date. On that date or later the work item will show a UI indication on the work item grid no matter what the current status.

global_email_date

This fields stores the global work item email date. On that date or later an e-mail will be sent to the owner on that work item no matter what the current status.

Both global_deadline_date and global_email_date actions are triggered by running a workflow utility that should run daily.

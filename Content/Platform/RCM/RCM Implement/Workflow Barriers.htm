Workflow Barriers

Workflows allow you to give specific validations for changing steps based on review rules.

Workflow barriers enable setting up rules that require one or more users to review a step before the transition is made. This can also be referred to as four-eye review. A review can only occur when a step is changed.

A step transition does not necessarily include a barrier, meaning it is not counted as reviewed.

The following terms are relevant for workflow step transitions:

  • Review: The change step action (on a transition with barriers).
  • Barrier: Definition of the number of required reviewers at that point needed to advance to the next step.
  • Barrier Weight: Represents the number of distinct users (that changed previous steps) required to move to the next step. If you define barrier 8 in a transition, it means you need 7 other users to have reviewed in previous steps in the workflow.

    If the other users changed steps without barriers on the transition then barrier weight=0 and those transitions were not considered reviewed.

  • Barrier weight 0: No barriers are defined for the transition, meaning it does not require review and is not counted as reviewed for the next transition. This is the default value for any transition.
  • Barrier weight 1: One person must review the step change.
  • Barrier weight 1+X: At least two people must review the step before it can be changed.

For details on barrier weight setup see Define Barriers.

Barrier Examples

All barriers are cumulative, that is if a barrier of two is provided to a transition then it is counted with the number of barriers for the previous transition.

The following examples describe two scenarios, the first where the workflow is correct and the second where it is incorrect.

Example 1

  • A barrier weight of 1 (one reviewer) is provided for the transition between Ready and In Process.
  • A barrier weight of 2 is provided between In Process and Open Issue.
  • If one person has reviewed between Ready and In Process, then only one more reviewer (different user) is needed to move from In Process to Open-Issue. Meaning the total number of reviewers for both step transitions is two.

Example 2

  • There is no barrier between the transition of Ready and Investigating (Barrier weight equals 0).
  • A user decides to move from Ready to Investigating, (Barrier weight equals 0).
  • The same user (or a different one) wants to move from Investigating and then to In Process (Barrier weight equals 0).
  • The same user (or different user) would not be able to move from In Process to Open-Issue no matter how many users review the transition between In Process and Open-Issue. This is due to the fact that the barrier is cumulative and no review was performed and counted previously. (0 is not counted as a review). In order to proceed, the implementer has to have set up at least one of the previous transitions in the workflow with barrier weight of 1.

You should carefully plan the barriers between transitions in order not to have situations in the workflow where a transition cannot be made since the cumulative barrier weight cannot equal the barrier weight in a specific transition.

Although the barrier workflow review logic is cumulative, when RCM calculates the number of reviewers, it refers to the last barriers needed for this calculation and not all the previous barriers that may have been defined.

Barrier Weight Scenarios

The following depicts possible workflow scenarios where different barrier weights are given in specific situations.

Scenario 1: Permitted

Total barrier weight equals 2 and the two users have reviewed and changed the step.

Scenario 2: Not permitted

Total barrier weight equals 2 and there is only one user available

Scenario 3: Permitted

Total barrier weight equals 1 so only User A is required for the workflow.

Scenario 4: Not permitted

Total barrier weight is 3 and there are not enough reviewers or step changes to allow this amount.

Scenario 5: Not permitted

Total barrier weight is 2 and there are two users, but the first transition requires no review so the barrier weight cannot be reached.

Scenario 6: Permitted

Total barrier weight is 3 and there are three users for the relevant step changes. (All previously reviewed steps are counted.)

Scenario 7: Permitted

Total barrier weight is 7 and there are seven users to review and change steps accordingly.

RCM Workflow

In the RCM web application, a step that still did not resolve its barrier requirements is greyed out and includes a tooltip that indicates that the step has review rules.

If the user attempts to make a transition to a step that did not resolve its barrier requirements via the quick change step action then the following message is displayed:

You cannot change to this step because of review rules.

Barriers do not modify the functionality of the workflow when the user recalls or resets a step.
